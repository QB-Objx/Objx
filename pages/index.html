Modeling

Use one model shape across type inference, hydration, and graph runtime.

Advanced column builders, default factories, codecs, naming strategy, and relation metadata are all part of the same core model API.

Model definition

Keep logical camelCase keys while mapping to physical database names, and still get runtime defaults, typed inserts, and graph-aware relations.

Typed model 
import { col, defineModel, hasMany } from '@qbobjx/core';
import { createSnakeCaseNamingPlugin } from '@qbobjx/plugins';

export const Project = defineModel({
  table: 'projects',
  columns: {
    id: col.bigInt().primary().default(() => generateSnowflakeId()),
    tenantId: col.text().generated(),
    budget: col.numeric(),
    profile: col.jsonb().nullable(),
    status: col.enum(['draft', 'active', 'done']).default('draft'),
    createdAt: col.timestamp(),
  },
  relations: (project) => ({
    tasks: hasMany(() => Task, {
      from: project.columns.id,
      to: Task.columns.projectId,
    }),
  }),
  plugins: [createSnakeCaseNamingPlugin()],
});

Postgres RLS

Tenant predicates are not the same thing as database-enforced RLS.

OBJX now supports a first-class bridge from request context into set_config(...) at transaction start. That means you can keep using PostgreSQL policies based on current_setting(...) without hand-wiring the same ceremony around every use case.

  • createTenantScopePlugin() handles SQL-level filtering in the ORM
  • executionContextSettings handles PostgreSQL session state for RLS
  • both can run together when you want app-level and DB-level enforcement
RLS transaction setup 
const executionContextManager = createExecutionContextManager();

const session = createPostgresSession({
  pool,
  executionContextManager,
  executionContextSettings: {
    bindings: [
      { setting: 'app.tenant_id', contextKey: 'tenantId', required: true },
      { setting: 'app.actor_id', contextKey: 'actorId' },
      {
        setting: 'request.jwt.claims',
        value: ({ executionContext }) => ({
          sub: executionContext.values.get('actorId'),
        }),
      },
    ],
  },
});

await executionContextManager.run(
  {
    values: {
      tenantId: 'tenant_a',
      actorId: 'user_123',
    },
  },
  () =>
    session.transaction(async (trx) => {
      return trx.execute(Project.query(), { hydrate: true });
    }),
);

NestJS

Use the official module, not a pile of custom providers.

@qbobjx/nestjs ships a dynamic module, typed session host, request context, and validation mapping so the database layer feels like part of the app, not an integration tax.

  • ObjxModule.forRoot(...) and forRootAsync(...)
  • ObjxSessionHost<TSession> for typed access without decorator noise
  • request values from headers already flow into the execution context
  • Postgres RLS can piggyback on the same request context through executionContextSettings
NestJS module setup 
@Module({
  imports: [
    ObjxModule.forRootAsync({
      global: true,
      useFactory: async () => ({
        session: createPostgresSession({
          pool,
          executionContextManager: createExecutionContextManager(),
          hydrateByDefault: true,
          executionContextSettings: {
            bindings: [
              { setting: 'app.tenant_id', contextKey: 'tenantId', required: true },
              { setting: 'app.actor_id', contextKey: 'actorId' },
            ],
          },
          plugins: [
            createTenantScopePlugin(),
            createSoftDeletePlugin(),
          ],
        }),
        requestContext: { enabled: true },
      }),
    }),
  ],
})
export class AppModule {}
